# ferromark 0.3 Integration Architecture

**Date:** 2026-07-11
**Status:** Approved
**Epic:** [#55](https://github.com/sebastian-software/ferromark/issues/55)

## Summary

ferromark 0.3 is one integration release delivered in one reviewable pull request. It removes Cargo features that do not represent real capabilities, adds an opt-in fenced-code rendering boundary to the Rust crate, publishes a native Node package, and moves the homepage to Ardo 4 with the shared `@ferramenta/ardo-config` package.

The implementation stays split into focused Conventional Commits so each child issue remains independently reviewable:

1. [#35](https://github.com/sebastian-software/ferromark/issues/35) — Cargo feature cleanup
2. [#33](https://github.com/sebastian-software/ferromark/issues/33) — fenced-code renderer
3. [#34](https://github.com/sebastian-software/ferromark/issues/34) — napi-rs distribution
4. [#38](https://github.com/sebastian-software/ferromark/issues/38) — Ardo 4 homepage

## Goals

- Make the published Cargo feature list match supported behavior.
- Let Rust callers replace fenced-code output without post-processing the complete HTML document.
- Preserve the existing streaming path and its allocation behavior when no custom renderer is installed.
- Publish an ESM-first Node package with generated TypeScript declarations and deterministic native loading.
- Provide a real ferromark and Ferriki integration without coupling either Rust core to the other.
- Use the shared Ferramenta site theme and family navigation on the public homepage.
- Keep Rust, Node, homepage, and release validation reproducible in CI.

## Non-goals

- WASM or browser distribution
- A direct dependency on Ferriki in the ferromark Rust crate
- A general callback system for arbitrary parser events
- Syntax highlighting inside indented code blocks
- Presentation features or syntax
- Unrelated dependency upgrades

## 1. Cargo feature contract

Remove `neon`, `trace`, and `std` from `Cargo.toml`:

- `neon` has no implementation and the Node package uses napi-rs.
- `trace` has no implementation.
- `std` is misleading because the crate does not support `no_std`.

Keep `mdx` as the only opt-in feature and declare an empty default feature set. The 0.3 migration guide will call out the removed names for consumers that explicitly enabled them. Removing a named feature is a Cargo compatibility break, so its commit uses `!`.

## 2. Rust fenced-code renderer

### Public API

The customization boundary is separate from `Options`. This keeps `Options` small, copyable, and on the existing fast path.

```rust
pub struct FencedCodeBlock<'a> {
    pub language: Option<&'a str>,
    pub code: &'a str,
}

pub struct TrustedHtml(String);

impl TrustedHtml {
    pub fn from_trusted(html: impl Into<String>) -> Self;
    pub fn as_str(&self) -> &str;
    pub fn into_string(self) -> String;
}

pub trait FencedCodeRenderer {
    fn render(&mut self, block: FencedCodeBlock<'_>) -> Option<TrustedHtml>;
}

pub fn to_html_with_renderer(
    input: &str,
    options: &Options,
    renderer: &mut dyn FencedCodeRenderer,
) -> String;

pub fn to_html_into_with_renderer(
    input: &str,
    output: &mut Vec<u8>,
    options: &Options,
    renderer: &mut dyn FencedCodeRenderer,
);
```

The exact Rust signatures may use the crate's existing writer error type where required, but the ownership, fallback, and trust semantics above are fixed.

### Inputs

- The hook runs only for fenced code blocks, never indented code blocks.
- `language` is the decoded first word of the CommonMark info string. It is `None` when the fence has no language.
- `code` is the raw code-block content after CommonMark block parsing, before HTML escaping.
- The renderer receives one complete fenced block because syntax highlighters need the complete source. Buffering happens only on the renderer-enabled fenced-code path.

### Output and trust boundary

`TrustedHtml` is written verbatim. Its `from_trusted` constructor is intentionally explicit: creating it asserts that the producer owns the HTML trust decision and has escaped all untrusted code, language, token, and diagnostic content.

The wrapper does not make unsafe HTML safe. Rust documentation and Node declarations must state that a custom renderer can introduce XSS if it returns attacker-controlled markup without escaping.

The hook remains available when `RenderPolicy::Untrusted` is selected. This is deliberate: the caller explicitly installs the renderer and therefore owns this separate trust boundary. The default fallback continues to escape code and language data under every policy.

### Fallback behavior

Returning `None` means "not handled". ferromark then emits its current escaped output:

```html
<pre><code class="language-rust">…</code></pre>
```

Unsupported languages and highlighter failures use this fallback. A renderer must not partially write output, so fallback cannot create mixed or malformed markup.

### Parser representation

The block event model must distinguish fenced and indented code blocks. A typed code-block kind is preferred over inferring the kind from a missing info string, because a fence without an info string is still fenced.

The default rendering entry points pass no renderer and keep the current streaming behavior. Only renderer-aware entry points hold an optional renderer reference and a reusable fenced-code buffer. Normal paragraphs and default code-block rendering do not allocate this buffer.

### Validation

- Unit tests for language extraction, raw code, missing languages, fallback, trusted output, and untrusted mode
- Regression tests proving indented code does not invoke the hook
- Tests proving fallback output matches the existing renderer byte-for-byte
- Parser benchmark guardrail with no renderer
- Code-heavy benchmark with fallback and custom rendering

## 3. Node distribution

### Workspace layout

Add a dedicated napi-rs package under `node/ferromark`. The root Rust crate remains free of napi dependencies; the Node crate depends on ferromark through the workspace path and builds as a `cdylib`.

The npm package name is `ferromark`, subject to a final registry availability check immediately before publishing. It is ESM-first and exports generated TypeScript declarations plus the JavaScript loader.

### Initial API

```ts
export interface Options {
  tables?: boolean
  strikethrough?: boolean
  taskLists?: boolean
  autolinks?: boolean
  // Remaining supported Rust options are mapped explicitly.
}

export function toHtml(markdown: string, options?: Options): string

export interface CodeHighlighter {
  codeToHtml(code: string, options: { lang?: string }): string
}

export function toHtmlWithHighlighter(
  markdown: string,
  highlighter: CodeHighlighter,
  options?: Options,
): string
```

The final `Options` interface maps every intentional public Rust option with the same defaults and naming. Unknown keys are rejected by TypeScript and ignored or rejected consistently at the native boundary; behavior is covered by tests.

`toHtmlWithHighlighter` calls the supplied synchronous `codeToHtml` method for fenced blocks. An initialized Ferriki highlighter satisfies this structural interface, which makes the example real without adding a Ferriki dependency. If the callback throws or the language is unsupported, the binding returns the normal escaped code block rather than partially rendered HTML. Fatal argument and native-loading errors remain explicit JavaScript errors.

Ferriki integration belongs at this layer because Ferriki's current Rust crate is a napi `cdylib`, not a reusable Rust library. Both projects therefore keep independent native cores while composing through their public Node APIs.

### Native loading and package contents

- Use explicit platform and architecture mappings.
- Emit a clear unsupported-platform error that includes the detected platform and architecture.
- Keep platform binaries in platform-specific optional packages or another deterministic napi-rs layout verified by clean-install tests.
- Pack only the ESM loader, declarations, README/license metadata, and intended native artifacts.
- Test both a locally packed package and a clean external consumer import.

### Versioning and publishing

Rust and npm releases share the same `0.3.0` version. Release Please remains the source of the release event. Native builds run for the intended release only, upload immutable artifacts, and publish through npm trusted publishing/OIDC without a long-lived npm token.

All third-party actions and reusable workflows must use immutable commit SHAs. Workflows receive only the minimum permissions and secrets needed by each job. Artifact download and package assembly validate expected names and fail on missing or duplicate native binaries.

## 4. Homepage integration

Upgrade the homepage from the pre-v4 Ardo component API to Ardo 4.1 or newer:

- replace `ArdoProvider`, `Layout`, `Header`, and `Footer` with the supported `ArdoRoot`, `ArdoHeader`, and `ArdoFooter` surface,
- add `@ferramenta/ardo-config`,
- import `@ferramenta/ardo-config/theme.css`,
- render `<FamilyLinks current="ferromark" />` in the footer,
- preserve guide navigation and the `/ferromark/` GitHub Pages base path,
- document the Rust and npm 0.3 quick starts, including the Ferriki composition example.

The Ferramenta package rename is already merged in `sebastian-software/ferramenta#3`. The homepage consumes the published package; it does not copy shared config source into this repository.

## 5. Error handling and security

- Default Markdown rendering remains untrusted and escaped.
- Renderer-provided `TrustedHtml` is an explicit caller-controlled trust boundary.
- Node callback exceptions produce deterministic fallback behavior and never expose partially generated markup.
- Native input conversion rejects invalid types before Rust rendering starts.
- Release jobs use least privilege, immutable dependencies, and npm OIDC.
- Packed-package tests prevent accidental publication of source trees, credentials, build caches, or unrelated workspace files.
- Homepage rendering does not use raw HTML injection for package-provided navigation.
